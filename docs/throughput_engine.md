# Batched Throughput Engine

CALO-RPD Studio v3.1 accelerates comparative evaluation and CALO policy training without changing the scientific formulation.

## Comparative evaluation

One persistent service owns each verified CUDA, Intel XPU, and CPU runtime. Independent optimizer runs execute concurrently inside the assigned service. Their synchronous population-evaluation calls enter a cross-run broker. Requests are merged only when their case checksum, scenario definitions, objective, robust aggregation, device, precision, and power-flow options match.

The broker concatenates candidate matrices, calls the common FP64 evaluator, then returns each run's ordered result slice. It does not alter random streams, populations, algorithm equations, evaluation counts, or stopping rules.

Before a campaign, valid normalized candidates are evaluated outside optimizer budgets at several microbatch sizes. The best stable candidate-evaluation throughput is stored in a JSON profile. Complete jobs and concurrency slots are then allocated according to measured evaluations per second. CPU/GPU memory and utilization settings remain admission safety limits.

## Policy training

Persistent CUDA/XPU actors and a persistent CPU rollout pool receive one identical policy snapshot per PPO epoch. Calibration episodes are timed and discarded. Fresh episode allocation can then follow measured complete actor transitions per second.

During explicit ORPD development stages, simultaneous actor episodes can submit compatible populations to the same cross-episode batch broker. Only fresh trajectories generated by the current snapshot enter PPO. A centralized PPO learner updates the candidate policy after all matching lanes return.

## Profiling

In-memory stage timing is stored with completed results for:

- mixed-variable decoding;
- scenario preparation;
- batched AC power flow;
- objective and constraint aggregation;
- robust result finalization;
- cross-run batching.

The profile is diagnostic, not a substitute for a controlled wall-clock benchmark.

## Scientific safeguards

- PyTorch FP64 remains the accelerated numerical precision.
- CPU/accelerator parity can be required before execution.
- Candidate-specific PV-to-PQ switching is isolated when bus-type sets diverge.
- Calibration evaluations are excluded from algorithm budgets.
- Final publication states are independently reconstructed by the trusted CPU reference.
- A frozen TEST campaign must verify `calo_v31_freeze.json` before execution.
